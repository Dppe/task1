# Detailed Technical Report: Vector Database Selection for AI‑Powered Frontend Code Generation

## 1. Executive Summary

We are building an automated system that translates user stories into production‑ready frontend code (data‑layer, config‑ui, react‑toolkit). A critical component is a **vector database** that retrieves the most relevant internal code examples to guide the AI model. After evaluating leading open‑source options against our specific requirements—hybrid search, metadata filtering, scalability, and Node.js integration—we recommend **Weaviate** as the primary choice, with **Qdrant** as a strong alternative.

**Key findings:**
- Weaviate offers **native hybrid search** (BM25 + vector), the simplest implementation path.
- Qdrant provides **superior metadata filtering** and lower cost, but requires sparse vectors for hybrid search.
- Both databases are production‑ready, horizontally scalable, and have mature SDKs.
- A proof‑of‑concept with Weaviate is the recommended next step.

---

## 2. System Requirements

Our retrieval system must satisfy:

| Requirement | Description |
|-------------|-------------|
| **Hybrid Search** | Combine semantic understanding (e.g., “login” matches authentication code) with exact keyword matching (e.g., filename `api.ts`). |
| **Metadata Filtering** | Fast filtering by `layer` (data‑layer, config‑ui, react‑toolkit), `framework` (React, Vue), `componentType` (api, slice, component), and other attributes. |
| **Scalability** | Support millions of code chunks with horizontal scaling. |
| **Latency** | <100 ms per query to keep the generation pipeline responsive. |
| **Production Readiness** | High availability, backups, monitoring, and easy operations (self‑hosted or managed). |
| **Integration** | First‑class Node.js/TypeScript SDK, compatibility with our embedding models and LLM pipeline. |

## 3. Candidate Overview

We evaluated six vector databases. The two leading open‑source contenders are:

| **Qdrant** | **Weaviate** |
|------------|--------------|
| Written in Rust, schema‑less, highly flexible | Written in Go, schema‑based, GraphQL API |
| Best‑in‑class metadata filtering (pre‑filtering) | Native hybrid search (BM25 + vector) out of the box |
| Supports sparse vectors for modern hybrid search | Built‑in modules for automatic vectorization |
| Slightly slower average query time (73.1 ms) | Faster average query time (51.7 ms) |
| Lower cost ($103/month for 1M 1536‑dim vectors) | Higher cost ($160/month for same) |
| Free tier: 1GB forever | Free tier: 14‑day trial |

## 4. Detailed Comparison

### 4.1 Hybrid Search

| Aspect | Qdrant | Weaviate |
|--------|--------|----------|
| **Approach** | Sparse vectors (e.g., SPLADE, BGE‑M3) + dense vectors | Native BM25 + vector with `alpha` tuning |
| **Implementation Effort** | Moderate (must generate and store two vector types) | Low (single call with `alpha` parameter) |
| **Quality** | Excellent – learned term weights often outperform BM25 | Excellent – well‑tuned BM25 is very effective for code |
| **Flexibility** | Can combine with ColBERT for token‑level matching | Fixed to BM25, but `alpha` allows easy tuning |

**Winner:** Weaviate for simplicity; Qdrant for advanced customization.

### 4.2 Metadata Filtering

| Aspect | Qdrant | Weaviate |
|--------|--------|----------|
| **Filter Types** | Match, MatchAny, Range, Geo, Nested, Boolean | Equality, Range, Geo, Nested, Boolean |
| **Performance** | Pre‑filtering via payload index – extremely fast | Post‑filtering – good, but can degrade under many filters |
| **Schema** | Schema‑less – add fields anytime | Schema required – properties must be defined upfront |

**Winner:** Qdrant – its filtering engine is the best in class.

### 4.3 Performance Benchmarks

| Metric | Qdrant | Weaviate |
|--------|--------|----------|
| **Avg Query Time** | 73.1 ms | 51.7 ms |
| **Filtered Search** | Excellent | Good |
| **Indexing Speed** | Fast | Fast |

**Winner:** Weaviate (raw speed), but both are well within our latency budget.

### 4.4 Integration & Ecosystem

| Aspect | Qdrant | Weaviate |
|--------|--------|----------|
| **Node.js SDK** | Good (REST/gRPC) | Excellent (GraphQL, intuitive queries) |
| **LangChain Support** | Yes | Yes |
| **Built‑in Vectorizers** | No (bring your own) | Yes (modules for OpenAI, Cohere, Hugging Face) |
| **Management Tools** | Qdrant Cloud, Helm charts | Weaviate Cloud, Helm charts |

**Winner:** Weaviate for developer experience and built‑in features.

### 4.5 Cost & Free Tier

| Aspect | Qdrant | Weaviate |
|--------|--------|----------|
| **Cost (1M 1536‑dim vectors)** | $103/month (self‑hosted free) | $160/month (self‑hosted free) |
| **Free Tier** | 1GB forever on Qdrant Cloud | 14‑day trial on Weaviate Cloud |

**Winner:** Qdrant for long‑term cost and free tier.

## 5. Decision Framework

We weighted each criterion according to our project priorities:

| Criterion | Weight | Qdrant Score | Weaviate Score | Rationale |
|-----------|--------|--------------|----------------|-----------|
| **Filtering on layer/framework** | 30% | 5 | 4 | Our three‑layer architecture demands fast, complex filtering. |
| **Hybrid search quality** | 25% | 4 | 5 | Both are excellent, but Weaviate’s native hybrid is simpler to tune. |

| **Scalability** | 10% | 5 | 5 | Both scale horizontally. |
| **Cost** | 10% | 5 | 3 | Qdrant is cheaper, especially at scale. |
| **SDK quality (Node.js)** | 5% | 4 | 5 | Weaviate’s GraphQL is more natural for web developers. |
| **Future flexibility** | 5% | 5 | 4 | Qdrant’s schema‑less design offers more adaptability. |
| **TOTAL** | 100% | **4.4** | **4.5** | **Weaviate slightly ahead** |

**Interpretation:** Weaviate edges out Qdrant primarily due to its simpler hybrid search implementation and faster query speed, which align with our goal of rapid development and responsive user experience. Qdrant remains an excellent choice if filtering complexity or long‑term cost becomes a primary concern.

## 6. Recommendation

### Primary Choice: **Weaviate**

**Why Weaviate fits our use case:**
- **Simplest hybrid search** – just set `alpha` (e.g., 0.7) to balance semantic and keyword matching; no need to generate and store sparse vectors separately.
- **GraphQL API** – intuitive for our Node.js/TypeScript team, with strong SDK support.
- **Faster queries** – 51.7 ms average ensures snappy retrieval.
- **Built‑in vectorization** – can optionally use modules to reduce pipeline complexity.
- **Batteries‑included** – modules for popular embedding models if we want to offload computation.

### When to Consider Qdrant Instead
- If our metadata filtering becomes extremely complex (e.g., deeply nested JSON, many high‑cardinality fields) and performance degrades.
- If we need the 1GB free tier for a long‑running prototype.
- If we prefer a schema‑less approach and are comfortable implementing sparse vectors for hybrid search.

## 7. Proposed Architecture with Weaviate

```
┌─────────────────────────────────────────────────────────┐
│                     User Story                           │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│                LLM Query Analyzer                        │
│  - Extracts intent, keywords, required layer/framework   │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│                  Retrieval Module                        │
│  ┌─────────────────┐    ┌───────────────────────────┐   │
│  │ Dense Embedding │    │ Metadata Filter Builder   │   │
│  │ (Voyage/BGE)    │    │ layer, framework, type    │   │
│  └────────┬────────┘    └─────────────┬─────────────┘   │
│           └──────────────┬────────────┘                  │
│                          ▼                               │
│              ┌─────────────────────┐                     │
│              │      Weaviate       │                     │
│              │ hybrid search       │                     │
│              │ alpha = 0.7         │                     │
│              │ + where filters     │                     │
│              └──────────┬──────────┘                     │
└──────────────────────────┼──────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────┐
│                 Top‑k Code Examples                      │
│  (content, file path, layer, score, metadata)           │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│                       Reranker                           │
│   (Cross‑encoder like Cohere Rerank or CodeBERT)        │
│   Improves precision of top results                      │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│                  Code Generator LLM                      │
│   Generates final frontend code using examples           │
└─────────────────────────────────────────────────────────┘
```

**Key integration points:**
- **Chunking:** Split code files using AST‑based chunking (by function, component, class) and attach rich metadata.
- **Embedding:** Use a code‑specific model (e.g., Voyage Code‑3, Snowflake Arctic Embed) to generate dense vectors.
- **Weaviate collection:** Define a `CodeExample` class with properties: `content` (text), `layer`, `framework`, `fileName`, `componentType`, `dependencies` (array), `createdAt` (date). Vectorization can be automatic (using a module) or manual.
- **Hybrid query:** Use `collection.query.hybrid()` with `alpha` and `where` filters.
- **Reranking:** Apply a cross‑encoder to the top 20 results, then pass the top 5 to the generator.
